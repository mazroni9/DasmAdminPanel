# DASM Admin Panel Security Baseline

هذه الوثيقة هي المرجع الأمني والتشغيلي الأساسي لمستودع DasmAdminPanel.

## الهدف

تثبيت قواعد إلزامية تمنع الخلط بين الكنترول روم المستقل وبين أي لوحة داخلية أخرى، وتمنع فتح صلاحيات أو مسارات أو تكاملات حساسة دون ضبط واضح.

### 1) هوية المستودع

هذا المستودع هو الكنترول روم المستقل الخاص بـ DASM.

لا يجوز الخلط بينه وبين DASM-Platform أو الكنترول روم الداخلي داخل المنصة.

أي تغيير يجب أن يوضح هل هو داخل هذا المستودع فقط أم يحتاج تنسيقًا مع مستودع آخر.

### 2) المسارات الرسمية

- المسار الرسمي للكنترول روم: `/admin/control-room`
- بوابة الدخول الرسمية: `/auth/login`
- `/dashboard` و`/login` يُعاملان كمسارات legacy حتى يتم تحويلهما أو إزالتهما صراحة.
- يمنع بناء أي features جديدة على المسارات القديمة.

### 3) قواعد الأمان العليا

هذا المستودع إداري وحساس.

لا يجوز منح وصول تشغيلي أو وصول لمراجعة الطابور أو إدارة الأعضاء لمستخدمين عاديين أو غير إداريين دون طلب صريح ومبرر.

لا تعتمد على حماية الواجهة فقط؛ التحقق الحقيقي يجب أن يكون من الخادم.

### 4) قواعد `pages/api/*`

أي route داخل `pages/api/*` يملك قدرة إدارية أو سرية أو يلامس مزودات خارجية يجب معاملته كمسار عالي الخطورة.

المسارات الاختبارية أو التشخيصية أو غير المستخدمة يجب تعطيلها أو إزالتها من الإنتاج.

أي route يستخدم service role أو عمليات إدارية على Supabase أو Cloudinary يجب أن يخضع لتحقق هوية وصلاحية واضح وصارم.

### 5) Supabase legacy

وجود أجزاء مرتبطة بـ Supabase لا يعني اعتمادها تلقائيًا.

أي تدفق legacy يبقى مجمدًا حتى يُحسم مصيره.

يمنع توسيع النظام القديم أو البناء فوقه قبل قرار معماري صريح.

### 6) إدارة الأسرار والبيئة

يمنع وضع أي secret داخل الكود أو ملفات tracked.

يمنع استخدام `NEXT_PUBLIC_*` لأي قيمة سرية.

build success لا يساوي readiness أمنية أو إنتاجية.

أي اعتماد على env أو domain أو Vercel أو CORS أو cookies أو refresh يجب توثيقه وفحصه صراحة.

### 7) قواعد التطوير

العمل دائمًا عبر branch ثم PR ثم merge.

ممنوع push مباشر إلى `master` إلا بطلب صريح.

أي تغيير يمس auth أو routes أو `pages/api` أو security أو domains أو env يحتاج أقل تعديل آمن ممكن مع تقرير واضح بالمخاطر.

### 8) قواعد legacy

لا يتم حذف legacy مباشرة إذا كان قد يؤثر على التشغيل.

المسار الصحيح هو: تجميد ثم redirect أو بديل ثم smoke test ثم إزالة لاحقة.

`/dashboard` و`/login` من أبرز أمثلة legacy التي يجب عدم البناء الجديد فوقها.

### 9) ما يجب فحصه قبل أي مهمة

- هل المشكلة في الكود أم في البيئة؟
- هل التعديل داخل DasmAdminPanel فقط أم يحتاج تنسيقًا خارجيًا؟
- هل هناك خطر على auth أو exposure أو users أو assets؟
- هل يوجد مسار legacy يجب عدم البناء فوقه؟

### 10) ما يجب تسليمه بعد أي مهمة

- ما الذي تغيّر
- الملفات المعدلة
- نوع التغيير: code / env / routing / security / docs
- هل التغيير آمن إنتاجيًا أم يحتاج smoke test إضافي
- ما الذي بقي risk أو follow-up

### الأولويات الأمنية الحالية

- حماية أو تعطيل أي `pages/api/*` حساسة أو legacy.
- توحيد الدخول الرسمي حول `/auth/login` ثم `/admin/control-room`.
- منع الاعتماد على `/dashboard` و`/login` القديمين كنقطة دخول تشغيلية.
- تقليل سطح الهجوم وإزالة المسارات الاختبارية من الإنتاج.
- منع التوسع غير المقصود في الوصول الإداري.

هذه القاعدة مبنية على المخاطر التي ظهرت في تقرير مسح الريبو، خصوصًا ازدواجية الدخول، وجود مسارات `pages/api/*` حساسة، واستمرار أجزاء legacy مرتبطة بـ Supabase داخل نفس الريبو.

**أهم نقطة تشغيلية:** هذا سيجعل Cursor، عند فتح الريبو، يقرأ قاعدة `alwaysApply` ويُعاملها كمرجع إلزامي، مثل قواعد `.cursor/rules` المعتادة في مشاريعكم الأخرى.
